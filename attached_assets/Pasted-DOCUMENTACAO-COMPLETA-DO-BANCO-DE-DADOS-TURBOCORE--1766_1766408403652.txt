DOCUMENTACAO COMPLETA DO BANCO DE DADOS - TURBOCORE
====================================================

VISAO GERAL
-----------
O TurboCore e uma aplicacao SaaS interna para gerenciamento de credenciais de clientes, ferramentas internas, clube de beneficios e base de conhecimento. Este documento descreve a estrutura completa do banco de dados com 5 tabelas principais.


DIAGRAMA DE RELACIONAMENTOS
---------------------------

                    USERS (usuarios)
                          |
        +-----------------+-----------------+-----------------+
        | created_by      | created_by      | created_by      |
        v                 v                 v                 v
    CLIENTS          COURSES          BENEFITS         TURBO_TOOLS
   (clientes)        (cursos)       (beneficios)      (ferramentas)
        |
        | client_id (1:N)
        v
   CREDENTIALS
  (credenciais)

Legenda:
- 1:N = Um para Muitos (um cliente tem varias credenciais)
- created_by = Referencia ao usuario que criou o registro


TIPOS ENUM (criar antes das tabelas)
------------------------------------

ENUM: course_status
Valores: ativo, vitalicio, cancelado, sem_status
Uso: Status do curso na tabela courses

ENUM: benefit_segment  
Valores: alimentos, beleza_cosmeticos, casa_cozinha, tecnologia, pet, plantas_agro, suplementacao, moda
Uso: Segmento do beneficio na tabela benefits

ENUM: recorrencia
Valores: Mensal, Anual
Uso: Frequencia de pagamento na tabela turbo_tools


========================================
TABELA: CLIENTS (Clientes)
========================================

Descricao: Armazena informacoes dos clientes da Turbo Partners. Cada cliente pode ter multiplas credenciais de acesso a diferentes plataformas.

Total de registros: 386

COLUNAS:

id
- Tipo: UUID
- Obrigatorio: Sim
- Padrao: gen_random_uuid()
- Descricao: Identificador unico do cliente

name
- Tipo: TEXT
- Obrigatorio: Sim
- Descricao: Nome do cliente ou empresa

cnpj
- Tipo: TEXT
- Obrigatorio: Nao
- Descricao: CNPJ do cliente (formato livre)

additional_info
- Tipo: TEXT
- Obrigatorio: Nao
- Descricao: Informacoes adicionais ou observacoes

created_by
- Tipo: UUID
- Obrigatorio: Sim
- Descricao: ID do usuario que cadastrou o cliente (FK para users)

created_at
- Tipo: TIMESTAMP
- Obrigatorio: Sim
- Padrao: now()
- Descricao: Data e hora de criacao do registro

updated_at
- Tipo: TIMESTAMP
- Obrigatorio: Sim
- Padrao: now()
- Descricao: Data e hora da ultima atualizacao

INDICES:
- clients_pkey: Chave primaria (id)

RELACIONAMENTOS:
- created_by referencia users.id
- clients.id e referenciado por credentials.client_id (um cliente tem N credenciais)

password
- Tipo: TEXT
- Obrigatorio: Nao
- Descricao: Senha de acesso

access_url
- Tipo: TEXT
- Obrigatorio: Nao
- Descricao: URL de acesso a plataforma

observations
- Tipo: TEXT
- Obrigatorio: Nao

created_by
- Tipo: UUID
- Obrigatorio: Sim
- Descricao: ID do usuario que cadastrou (FK para users)

created_at
- Tipo: TIMESTAMP
- Obrigatorio: Sim
- Padrao: now()
- Descricao: Data e hora de criacao

updated_at
- Tipo: TIMESTAMP
- Obrigatorio: Sim
- Padrao: now()

========================================
ORDEM DE IMPORTACAO
========================================

Para importar os dados corretamente, siga esta ordem:

1. Criar ENUMs (course_status, benefit_segment, recorrencia)
2. Tabela users (se necessario - referenciada por created_by)
3. Tabela clients (independente, apenas referencia users)
4. Tabela credentials (depende de clients)
5. Tabela courses (independente, apenas referencia users)
6. Tabela benefits (independente, apenas referencia users)
7. Tabela turbo_tools (independente, apenas referencia users)

========================================
OBSERVACOES DE SEGURANCA
========================================

1. SENHAS EM TEXTO: As senhas estao armazenadas em texto puro. Considere implementar criptografia no novo sistema.

2. UUIDS PRESERVADOS: Os IDs originais foram mantidos para preservar os relacionamentos entre tabelas.

3. CAMPO CREATED_BY: Referencia usuarios do sistema original. Ajuste conforme a estrutura de usuarios do novo sistema ou remova a restricao de FK temporariamente durante a importacao.

4. AUDITORIA: O sistema original possui logs de auditoria para acoes sensiveis (visualizacao e copia de senhas). Considere implementar no novo sistema.
